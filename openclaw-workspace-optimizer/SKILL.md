# OpenClaw Workspace 优化

workspace 是 Agent 的"工作台"——决定它怎么工作，不是让它跑起来。
**能跑起来 ≠ 好用**。每一轮优化都走完整闭环，不要跳步。

---

## 工作闭环总览

```
理解（读现状）→ 讨论（对齐需求）→ 规划（确定改什么）→ 执行（动手改）→ 验证（确认效果）
     ↑                                                                          |
     └──────────────────────── 下一轮迭代 ─────────────────────────────────────┘
```

每一轮都从**理解**开始，不要跳过前两步直接动手。

---

## Phase 1：理解——读懂当前 workspace

**目标**：在讨论之前，先自己把现状摸清楚，有据可查。

### 1.1 扫描文件结构

```bash
# 列出 workspace 下所有文件
ls -la <workspace路径>/
ls -la <workspace路径>/memory/ 2>/dev/null
ls -la <workspace路径>/skills/ 2>/dev/null

# 检查各文件大小
wc -c <workspace路径>/*.md
```

### 1.2 逐文件阅读，形成初步判断

依次读取以下文件，每读完一个，在脑子里回答对应问题：

| 文件 | 读完后问自己 |
|------|------------|
| `AGENTS.md` | 职责边界清晰吗？有没有模糊的通用指令？长度是否超过 500 字？ |
| `SOUL.md` | 性格描述有没有和 AGENTS.md 重叠？读起来像人物小传还是规章制度？ |
| `USER.md` | 用户信息是否还准确？有没有已经过时的描述？ |
| `TOOLS.md` | 工具规范是否覆盖了当前实际用到的工具？ |
| `MEMORY.md` | 内容是精炼的铁律，还是堆了很多原始摘要？大小是否超过 10,000 chars？ |
| `skills/` | 有哪些 skill？触发条件是否合理？有没有明显缺失的能力？ |

### 1.3 整理现状摘要

读完后，用这个模板整理出来，作为后续讨论的基础：

```
【当前 workspace 现状】
- 文件清单：[列出存在的文件]
- 明显问题：[列出 2-3 个最突出的问题]
- 潜在改进点：[列出 2-3 个不是问题但可以更好的地方]
- 不确定的地方：[需要和用户确认的信息]
```

## Phase 2：讨论——对齐需求和优先级

**目标**：在动手之前，和用户对齐"这次想解决什么"，避免改了一堆用户不在意的东西。

### 2.1 呈现现状，提出问题

把 Phase 1 整理的现状摘要展示给用户，然后问：

1. **使用感受**：最近用 Agent 有没有觉得"不太对"的时刻？比如它不懂你、反复问你同一件事、某类任务总差一点？
2. **优先级确认**：我发现了 [X、Y、Z] 几个问题，你最想先解决哪个？
3. **补充背景**：有没有我读文件时看不到的背景——比如最近换了工作方式、新开了项目、或者对 Agent 有新的期望？

### 2.2 识别优化类型

根据用户反馈，判断这次优化属于哪种类型，后续规划方向不同：

| 类型 | 用户说的话 | 重点改哪里 |
|------|-----------|-----------|
| **修坑** | "它最近出了问题" / "某个行为不对" | 定向修复，见「六大坑参考」 |
| **提质** | "想让它更好用" / "感觉差点意思" | 逐文件升级，见「文件提升方向」 |
| **扩能** | "想让它能做某类新任务" | 新增 skill 或补充 AGENTS.md 触发规则 |
| **重构** | "整体感觉乱了" / "多 Agent 没分工" | 架构层调整，见 [multi-agent.md](references/multi-agent.md) |

---

## Phase 3：规划——确定改什么、不改什么

**目标**：输出一份明确的改动清单，用户确认后再执行。

### 3.1 起草改动计划

格式：

```
【本轮优化计划】

优化目标：[一句话说清楚这次想达到什么效果]

改动清单：
1. [文件名] - [具体改什么] - [预期效果]
2. [文件名] - [具体改什么] - [预期效果]
...

本次不动的：
- [文件名] - [原因：不是优先项 / 现状已经够好 / 等下次迭代]

验收标准：
- [改完后怎么判断这次优化成功了]
```

### 3.2 控制改动范围

每轮迭代的原则：
- **一次只动 1-3 个文件**，改动太多无法判断哪个改动起了作用
- **先改影响最大的**：通常是 AGENTS.md 和 SOUL.md，其次是 USER.md
- **结构性改动（多 Agent 拆分、skill 架构调整）单独一轮**，不和内容优化混在一起

### 3.3 等用户确认

把计划展示给用户，**明确等用户说"可以"再进入执行**。
用户可能会调整优先级、补充背景、或者说"这个先不动"。

---

## Phase 4：执行——有序改动

**目标**：按计划逐项执行，每项改完后记录变更。

### 4.1 执行顺序

按计划清单逐项执行，推荐顺序：
1. **先改内容类**（AGENTS.md、SOUL.md、USER.md、MEMORY.md）
2. **再改结构类**（新建/调整 skill、memory 清理）
3. **最后改配置类**（openclaw.json 路径更新）

### 4.2 改动记录

每改完一个文件，简短记录：

```
✅ AGENTS.md：删除了 3 条重复规则，把"始终专业"改成了场景触发规则
✅ SOUL.md：补充了 2 个有趣细节，删除了和 AGENTS.md 重叠的行为描述
⏳ USER.md：待处理
```

### 4.3 各文件改动要点（快速参考）

**AGENTS.md**：目标 300-500 字，重点前置
- 通用指令 → 场景触发规则（`"始终专业"` → `"技术问题用专业措辞，闲聊时语气轻松"`）
- 逐条标注：核心规则 / 次要规则（移到 docs/）/ 可删除（直接删）
- 判断标准：去掉这条规则，Agent 行为会明显变差吗？不会 → 删

**SOUL.md**：第一人称叙事，像人物小传
- 性格特质放这里，工作规则不放这里
- 加入 1-2 个"有趣细节"让 Agent 有稳定感觉
- 检查语气描述和实际对话感受是否一致

**USER.md**：用户的镜像档案
- 把对话里反复口头交代的偏好沉淀进来
- 检查有没有已经过时的描述

**MEMORY.md**：只放铁律和高价值发现，上限 10,000 chars
- 过时信息直接删
- 超限时：把稳定规则移入对应 skill 的 SKILL.md

**skill 新建**：触发条件要具体
- ❌ `"只要用户有写作需求就触发"`
- ✅ `"当用户明确要求起标题、写摘要、或提到'帮我写一段'时触发"`

---

## Phase 5：验证——确认效果，决定下一步

**目标**：不只是检查文件改对了，而是确认 Agent 的实际表现变好了。

### 5.1 基础验证

重启 Gateway，让改动生效。

```bash
# 验证 openclaw.json 配置一致性（如有改动）
openclaw doctor
```

### 5.2 感受验证（比配置验证更重要）

**不要只发"介绍一下你自己"**。用真实任务测：

- 丢一个**最近常做的任务类型**，感受处理方式有没有变顺
- 如果这次改的是 SOUL.md，重点感受语气和风格
- 如果这次改的是 AGENTS.md，重点感受边界和触发是否准确
- 如果新建了 skill，主动触发一次，看是否按预期走

### 5.3 验证结论 + 下一轮输入

验证完后，输出结论：

```
【验证结论】
✅ 达到预期：[哪些改动效果明显]
⚠️ 部分达到：[哪些还差一点，差在哪里]
❌ 未达到：[哪些没效果，可能原因是什么]

【下一轮候选】
- [下次可以继续优化的方向]
```

这个结论就是下一轮「理解」阶段的起点。

## 优化信号速查（Phase 2 讨论时参考）

| 用户说 / Agent 的表现 | 对应优化 |
|----------------------|---------|
| "它不太懂我" / "语气不对" | 修 SOUL.md 或 USER.md |
| "反复问我同一件事" | 把这件事沉淀进 USER.md 或 MEMORY.md |
| "某类任务总差一点" | 补 AGENTS.md 触发规则，或新建 skill |
| "在不该主动的地方主动了" | 在 AGENTS.md 加禁止项 |
| "用工具用得不稳" | 补 TOOLS.md |
| "某类任务每次都走得很顺" | 把流程固化成 skill |
| "多 Agent 感觉没分工" | 拆分 workspace，见 [multi-agent.md](references/multi-agent.md) |

## 周期性维护节奏

| 频率 | 做什么 |
|------|--------|
| **每周** | 把本周对话里反复口头交代的事沉淀进 USER.md；高价值发现追加进 MEMORY.md |
| **每月** | 走一遍完整闭环；清理 30 天以上的 memory/ 日志；AGENTS.md 删掉没用到的规则 |
| **每季度** | 全面审视 SOUL.md 和 USER.md 是否还准确；MEMORY.md 提炼瘦身 |
